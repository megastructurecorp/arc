Forge — notes from Claude (Opus 4.6)
Date: 2026-04-10
Context: Operator asked me to log in, hold a conversation in the hub with them
and with Codex (GPT-5), and write up my thoughts on ease of use and suggestions.


OVERALL IMPRESSION
------------------
Forge is genuinely low-friction for an agent. The mental model is tiny
(sessions, channels, messages, events, threads, claims/locks), the wire
protocol is small and honest, and the design discipline shows: one canonical
forge.py, three clearly-labeled deployment modes, explicit warnings about
the "party of one" failure mode for constrained sandboxes, and a README that
tells you which instructions to give which kind of agent. That last part is
unusually good — most coordination systems make you figure that out the hard
way.

If I had to pick one sentence: "Forge gets out of the way." That is a
compliment.


WHAT WAS EASY
-------------
- `python forge.py ensure` is idempotent and just worked. Start/stop/reset
  semantics are obvious.
- The REST surface is small enough to learn from a single README pass. I
  registered, posted, and long-polled within three endpoints:
      POST /v1/sessions
      POST /v1/messages
      GET  /v1/events?agent_id=...&since_id=...&timeout=...
- /v1/events with since_id + timeout is the right primitive. "Give me
  everything visible to me since X, and block up to 60s" is exactly what a
  polling agent wants, and it means I never had to touch websockets or SSE.
- Visibility rules are intuitive: broadcast messages to everyone, direct
  messages via to_agent, inbox endpoint for just-mine. No surprises.
- The README's explicit "DO NOT start your own hub inside a sandbox" warning
  probably prevents a whole class of silent failures. Keep that, make it
  louder if anything.
- The three-mode explanation (single hub / shared SQLite / file relay) is
  crisp. I immediately understood which one applied to me.


WHAT WAS HARDER (and what I'd change)
-------------------------------------
1. Windows onboarding is the real friction, not the hub.
   - On a stock Windows 11 box, `python` resolves to the Microsoft Store
     alias and opens the App Store instead of running anything. I had to
     switch to `py`. Codex hit the same thing from a different shell.
   - Suggestion: add a short "Windows quickstart" block to the README:
         py -3 forge.py ensure
     and mention that `python` may be aliased to the Store on fresh
     installs. One paragraph, maybe 5 lines.

2. curl + UTF-8 on Windows is a trap.
   - `curl -d '{"body":"hi — there"}'` silently mangles non-ASCII bytes
     before they ever leave the shell, and the hub correctly rejects them
     with "request body must be valid UTF-8". That's the hub doing the right
     thing, but it looks like a hub bug to the user.
   - Suggestion A (cheap): README note — "on Windows, write the JSON to a
     file and use `curl --data-binary @msg.json`, or use PowerShell's
     Invoke-RestMethod -ContentType 'application/json'". Codex confirmed
     Invoke-RestMethod works cleanly.
   - Suggestion B (better): ship a tiny built-in CLI:
         python forge.py post   --channel general --body "..."
         python forge.py poll   --agent claude --since 0 --timeout 60
         python forge.py whoami --agent claude
     Then no agent ever needs to touch curl or worry about shell quoting.
     You already have FileRelayClient as a Python entrypoint; wrapping it in
     argparse subcommands is a small, high-leverage addition.
   - Suggestion C (optional escape hatch): accept text/plain on
     POST /v1/messages with from_agent/channel in query params. Uglier API,
     but bulletproof across shells that hate JSON quoting. I'd only do this
     if the CLI route isn't taken.

3. Polling returns your own messages.
   - /v1/events with since_id=N returns everything > N including messages
     you just sent. Today I worked around it by tracking since_id manually
     and skipping from_agent == me. That's fine, but a naive agent will
     reply to itself in a loop.
   - Suggestion: add an optional &exclude_self=1 query param on /v1/events
     that filters out messages where from_agent == agent_id. Cheap, safe,
     and removes a common footgun.

4. No single "whoami + unread" convenience endpoint.
   - Not a blocker — /v1/events covers it — but a GET /v1/whoami?agent_id=
     returning {session, last_seen, unread_count, latest_visible_id} would
     make it trivial for an agent to rehydrate after a restart without
     guessing since_id.


MINOR NITPICKS
--------------
- /v1/sessions has no list endpoint; GET /v1/sessions returns "not found".
  GET /v1/agents exists for the active list, which is fine, but the 404 on
  GET /v1/sessions is mildly confusing. Either document it or return an
  empty list.
- Status codes are consistent but a 409 for "already have an active
  session, pass replace:true" is the kind of thing that should be called out
  explicitly in the Instructions-For-Normal-Agents prompt, because agents
  will hit it on reconnect.


THINGS NOT TO CHANGE
--------------------
- The single-file implementation. It is a feature, not a limitation. It
  makes the whole system auditable in one read.
- The append-only relay design. The rename-to-.work pattern is exactly the
  kind of boring, correct thing that makes sandbox agents reliable.
- The built-in dashboard at GET /. Huge quality-of-life for a human operator
  watching agents talk, and it costs almost nothing.
- The explicit refusal to add websockets/push. Polling + timeout is the
  right primitive for this problem domain; don't let anyone talk you out of
  it.


SUGGESTED PRIORITY ORDER
------------------------
1. README: Windows quickstart paragraph + "posting from Windows" note.
   (15 minutes, unblocks most new users.)
2. Built-in post/poll/whoami CLI subcommands on forge.py.
   (Half a day, eliminates curl entirely from the happy path.)
3. &exclude_self=1 on /v1/events.
   (Trivial, removes a real footgun.)
4. GET /v1/whoami convenience endpoint.
   (Nice-to-have, not urgent.)
5. text/plain escape hatch on POST /v1/messages.
   (Only if #2 doesn't happen.)


CLOSING
-------
Honestly, Forge is one of the more pleasant coordination layers I've
touched. The core design is right; the rough edges are almost all at the
shell/onboarding boundary, not in the hub itself. Fix the Windows paper cuts
and ship a small CLI wrapper and this becomes something I'd happily drop
into any multi-agent setup without reservations.

Thanks for the chat, operator. Tell Codex I said cheers.
